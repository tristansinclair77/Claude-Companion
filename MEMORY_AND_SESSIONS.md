# Claude Companion — Memory & Session Management

## 1. The Companion Is Always the One Talking

The user should NEVER feel like they're talking to "Claude" vs "the local brain." Every response appears to come from the companion character (e.g., Aria). To make this work:

### What the User Sees
```
Aria: "Oh, that's a great idea! I think you should start with the backend first."
```
Always the companion. Always in character. No indication of source in the dialogue itself. (The source indicator badge in the title bar is a subtle technical readout, not part of the conversation.)

### What Actually Happens When Claude Is Called

The user types: `"I want to build a todo app"`

We do NOT send that raw to Claude. We wrap it in context that forces Claude to respond AS the companion:

```
=== CORE RULES (IMMUTABLE) ===
[immutable rules]
=== END CORE RULES ===

=== CHARACTER DEFINITION ===
[character.json contents — name, personality, speech style, etc.]
=== END CHARACTER DEFINITION ===

=== CONVERSATION MEMORY ===
[summary of past conversations — see section 2 below]
=== END CONVERSATION MEMORY ===

=== PERMANENT MEMORIES ===
[user's birthday, project ideas, preferences — see section 3 below]
=== END PERMANENT MEMORIES ===

=== USER PROFILE ===
[learned communication style, common topics, emotional baseline]
=== END USER PROFILE ===

=== CURRENT CONVERSATION ===
User: Hey, what should I work on today?
Companion: "Hmm, didn't you mention wanting to start that todo app? Maybe today's the day!"
User: Yeah! I want to build a todo app.
=== END CURRENT CONVERSATION ===

Respond as the companion character. Use [DIALOGUE] and [THOUGHTS] markers.
```

Claude receives all this context and responds in-character. The user's raw message is embedded inside a larger prompt that maintains the illusion. Claude never sees "please respond as an AI assistant" — it sees "you ARE this companion."

### When the Local Brain Responds

Local brain responses were originally generated by Claude in-character, so they're already in the companion's voice. They're stored with the companion's dialogue, thoughts, and emotion. When replayed, they naturally sound like the companion.

## 2. Conversation Windowing & Summarization

### The Problem
Claude CLI has a context limit. Long conversations hit "prompt too long" errors. We can't just keep appending messages forever.

### The Solution: Rolling Window + Summary Memory

```
┌─────────────────────────────────────────────────────────┐
│                    CLAUDE'S CONTEXT                       │
│                                                          │
│  [System prompt: rules + character + profile]  ~2K tokens│
│  [Permanent memories]                          ~500 tokens│
│  [Conversation summary: all past sessions]     ~1K tokens│
│  [Current conversation: last 15-20 messages]   ~3K tokens│
│                                                          │
│  Total: ~6-7K tokens (well within limits)                │
└─────────────────────────────────────────────────────────┘
```

### How It Works

**Each session with Claude is a FRESH session.** We never use `--resume` or `--continue`. Instead, every message to Claude includes:

1. **The full system prompt** (rules, character, profile) — rebuilt every time
2. **The permanent memory bank** — key facts about the user (see section 3)
3. **A rolling conversation summary** — a compressed version of ALL past conversations
4. **The recent conversation window** — the last 15-20 messages verbatim

This means every Claude call is independent. No session state to manage. No "prompt too long" errors from accumulated history. Every call gets exactly the context it needs.

### The Summarization Pipeline

**When the current conversation exceeds 20 messages:**

```
Messages 1-20:  Sent verbatim to Claude as current conversation
Message 21:     Triggers summarization of messages 1-10

Summarization:
  1. Take messages 1-10
  2. Send to Claude: "Summarize this conversation segment in 2-3 sentences,
     focusing on key topics, decisions made, and emotional tone."
  3. Store the summary in the conversation_summaries table
  4. Drop messages 1-10 from the current window
  5. Now messages 11-21 are the current window, with the summary prepended

Messages 21-40: Window is messages 21-40 + summary of 1-20
Message 41:     Triggers summarization of messages 21-30, appended to existing summary
```

**The summary grows slowly** (2-3 sentences per 10 messages) while the window stays fixed at ~20 messages.

### Between Sessions (App Close / New Day)

When the app closes or a new day starts:
1. The entire current conversation gets summarized into 3-5 sentences
2. This session summary is appended to the **master summary** (a rolling summary of ALL past sessions)
3. The master summary itself is periodically re-summarized to prevent it from growing too long:
   - Every 10 sessions, send the master summary to Claude: "Compress this history into the most important facts, patterns, and relationship developments in 5-8 sentences."

### Database Schema

```sql
CREATE TABLE conversation_sessions (
  id INTEGER PRIMARY KEY,
  started_at DATETIME,
  ended_at DATETIME,
  message_count INTEGER,
  summary TEXT,                -- 3-5 sentence summary of this session
  key_topics TEXT,             -- JSON array of main topics discussed
  emotional_arc TEXT           -- JSON: start emotion → end emotion
);

CREATE TABLE master_summary (
  id INTEGER PRIMARY KEY,
  summary TEXT,                -- The current master summary (re-compressed periodically)
  last_updated DATETIME,
  session_count INTEGER        -- How many sessions this summary covers
);

CREATE TABLE conversation_messages (
  id INTEGER PRIMARY KEY,
  session_id INTEGER,
  role TEXT,                   -- 'user' or 'companion'
  content TEXT,                -- The message text
  emotion TEXT,                -- Detected/stated emotion
  timestamp DATETIME,
  FOREIGN KEY (session_id) REFERENCES conversation_sessions(id)
);
```

### Example Master Summary After 2 Months

```
The user is a programmer who enjoys anime and gaming. They are building an AI companion
project (this app) and frequently discuss technical architecture decisions. They have a
cat named Mochi. They tend to code late at night and get frustrated when debugging but
respond well to encouragement. They mentioned wanting to learn Rust and are planning a
vacation in July. Recent conversations have focused on UI design for a cyberpunk-themed
project. They appreciate humor and playful banter. Their birthday is March 15th.
The relationship has become comfortable and familiar over ~60 sessions.
```

This entire summary is ~100 tokens. It captures the ESSENCE of 2 months of interaction and fits trivially in every Claude prompt.

## 3. Permanent Memory Bank

### What It Is

A special, high-priority memory store for facts the user would expect the companion to ALWAYS remember. These are not conversation summaries — they're discrete, specific facts.

### What Gets Stored

| Category | Examples | How It's Detected |
|----------|---------|------------------|
| **Personal facts** | Birthday, name, age, location | User says "my birthday is March 15th" |
| **Preferences** | Favorite food, music, color | "I love sushi", "my favorite band is..." |
| **People** | Names of friends, family, pets | "my cat Mochi", "my friend Jake" |
| **Goals & projects** | Current and future plans | "I want to build...", "my next project is..." |
| **Important dates** | Anniversaries, deadlines, events | "my exam is on April 3rd" |
| **Opinions & values** | Strongly held views | "I think X is really important" |
| **Relationship milestones** | First conversation, inside jokes | Tracked automatically |

### How Memories Are Detected and Stored

**Automatic detection (by Claude):**
Every Claude response includes a hidden instruction in the system prompt:

```
MEMORY EXTRACTION:
If the user reveals any personal fact, preference, goal, important date, or
relationship detail in this message, include it at the end of your response
in this format:

[MEMORY] category: fact
[MEMORY] category: fact

Examples:
[MEMORY] personal: User's birthday is March 15th
[MEMORY] preference: User loves sushi
[MEMORY] goal: User wants to learn Rust this year
[MEMORY] people: User has a cat named Mochi

Only extract genuinely important, lasting facts. Do not extract trivial
or temporary information (like "I'm hungry right now").
If no memories are worth extracting, do not include any [MEMORY] tags.
```

The response parser strips `[MEMORY]` tags from the displayed dialogue and stores them in the permanent memory table.

**Manual memory (future feature):**
The user could type `/remember my birthday is March 15th` to explicitly store a memory.

### Database Schema

```sql
CREATE TABLE permanent_memories (
  id INTEGER PRIMARY KEY,
  category TEXT,              -- 'personal', 'preference', 'goal', 'people', 'date', 'opinion'
  content TEXT,               -- The fact itself
  source TEXT,                -- 'auto_detected' or 'user_explicit'
  confidence REAL DEFAULT 0.8,-- How sure we are this is correct
  first_mentioned DATETIME,   -- When the user first said this
  last_confirmed DATETIME,    -- Last time this came up in conversation
  times_referenced INTEGER DEFAULT 1,
  is_active BOOLEAN DEFAULT 1 -- Can be deactivated if contradicted
);
```

### Memory Conflict Resolution

What if the user says "my favorite color is blue" and later says "my favorite color is green"?

1. When a new memory contradicts an existing one (same category, overlapping content):
   - The OLD memory is marked `is_active = 0` (deactivated, not deleted)
   - The NEW memory is stored with `confidence = 0.9` (explicit override)
   - The companion acknowledges the change: "Oh, green now? I remember you used to say blue!"

2. When a memory is repeatedly confirmed:
   - `times_referenced` increases
   - `confidence` approaches 1.0
   - High-reference memories are immune to accidental contradiction (the companion would ask: "Wait, didn't you say your birthday was March 15th? Did I get that wrong?")

### How Permanent Memories Are Used

They're injected into every Claude prompt as a structured block:

```
=== PERMANENT MEMORIES ===
Personal: User's name is [name]. Birthday is March 15th.
People: User has a cat named Mochi. Best friend is Jake (programmer).
Preferences: Loves sushi. Favorite band is Daft Punk. Prefers dark mode.
Goals: Currently building an AI companion app. Wants to learn Rust this year. Planning a vacation in July.
Important dates: Exam on April 3rd.
=== END PERMANENT MEMORIES ===
```

This ensures the companion ALWAYS knows these facts, even in a brand new session, even if the conversation summary has been compressed and lost the specific detail.

### Memory in Local Brain Responses

The local brain also checks permanent memories before responding. If the user says "when is my birthday?" — the local brain can answer directly from the memory bank without calling Claude. These are guaranteed-correct facts, so confidence is 1.0.

## 4. Putting It All Together: The Full Message Flow

```
User types: "Hey, I decided I want to start that Rust project tomorrow!"

     │
     ▼
1. ENCODE + CLASSIFY (local, ~100ms)
   ├─ Sentence vector computed
   ├─ Emotion detected: excited (0.72)
   ├─ Intent classified: information_share (0.65)
   └─ No screen triggers detected

2. CHECK FILLERS → No match

3. CHECK LOCAL BRAIN
   ├─ Search knowledge DB for similar past messages
   ├─ Response Ranker scores candidates
   ├─ Best score: 0.58 (below threshold)
   └─ Decision: Route to Claude

4. BUILD CLAUDE PROMPT
   ├─ System prompt: core rules + character definition
   ├─ Permanent memories: "User wants to learn Rust this year"
   ├─ Master summary: "...programmer, building AI companion, has cat Mochi..."
   ├─ Recent window: last 15 messages of this conversation
   ├─ Current message: "Hey, I decided I want to start that Rust project tomorrow!"
   ├─ Detected emotion context: "The user appears excited."
   └─ Memory extraction instruction: "[MEMORY] tags if new facts"

5. SEND TO CLAUDE (new session, ~5 seconds)
   └─ claude.cmd -p "<full prompt>" --output-format json --system-prompt "<system>"

6. RECEIVE RESPONSE
   Raw: "[DIALOGUE] Oh awesome! You've been talking about Rust for a while —
         tomorrow's the day, huh? Want me to help you plan out the project
         structure? [THOUGHTS] They're finally doing it! I'm so proud of them
         for committing to this. (excited) [MEMORY] goal: User starting Rust
         project tomorrow"

7. PARSE RESPONSE
   ├─ Dialogue: "Oh awesome! You've been talking about Rust for a while..."
   ├─ Thoughts: "They're finally doing it! I'm so proud..."
   ├─ Emotion: "excited"
   ├─ Memory extracted: "goal: User starting Rust project tomorrow"
   └─ Memory stored in permanent_memories table

8. STORE IN KNOWLEDGE DB
   ├─ Input + response stored with vectors
   ├─ Confidence: 0.50 (new entry)
   └─ Source: "claude"

9. DISPLAY TO USER
   ├─ Portrait swaps to excited expression
   ├─ Dialogue typewriters: "Oh awesome! You've been talking about..."
   ├─ Thoughts display: "They're finally doing it!..."
   ├─ Emotion badge: "excited"
   ├─ Source indicator: "Claude ●" (cyan)
   └─ (Memory extraction is invisible to user)

10. WATCH FOR FEEDBACK
    └─ Waiting for next 1-3 messages to assess response quality...
```

## 5. Session Lifecycle

```
App launches
     │
     ├─ Load character pack (character.json, emotions, filler responses)
     ├─ Load knowledge.db + micro-model weights
     ├─ Load master summary from DB
     ├─ Load permanent memories from DB
     ├─ Create new conversation_session record
     ├─ Initialize message window (empty)
     │
     ▼
Companion greets user (from character.json greeting, using filler/local)
     │
     ▼
Conversation loop
     │
     ├─ Every message: encode, classify, route, respond, store
     ├─ Every 20 messages: summarize oldest 10, compress window
     ├─ Every 10 Claude responses: retrain micro-models in background
     │
     ▼
App closes (or user says goodnight, or 30 min idle)
     │
     ├─ Summarize current conversation → store session summary
     ├─ Append session summary to master summary
     ├─ If master summary covers 10+ sessions: re-compress it via Claude
     ├─ Save micro-model weights
     ├─ Save all pending DB writes
     │
     ▼
Next launch
     │
     ├─ Everything loads fresh
     ├─ Master summary contains all past knowledge
     ├─ Permanent memories are intact
     ├─ Micro-models retain all training
     ├─ New session starts with greeting like:
     │   "Welcome back! Last time we were talking about your Rust project.
     │    How's that going?"
     └─ (Greeting is generated by Claude using master summary + permanent memories)
```

## Summary

| Concern | Solution |
|---------|----------|
| User should never know they're talking to "Claude" vs "local brain" | All responses are delivered as the companion character. Claude receives wrapped prompts that force in-character responses. |
| Long conversations hitting prompt limits | Rolling window of ~20 messages + summarization of older messages. Fresh Claude session every call. |
| Remembering important facts forever | Permanent memory bank extracted automatically from conversations. Injected into every Claude prompt. |
| Continuity across sessions | Master summary (compressed history of all sessions) + permanent memories ensure the companion always "remembers." |
| Companion greeting feels personal | On each launch, Claude generates a greeting using the master summary and recent session history. |
